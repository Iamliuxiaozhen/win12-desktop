# 维护者守则

本守则适用于拥有本仓库管理、发布、工作流修改、Issue 管理等权限的维护者。维护者在使用权限时，应当优先保证项目记录完整、决策过程可追溯，并避免绕过公开讨论直接改变项目状态。

## 基本原则

1. 影响用户获取、安装、更新或验证项目的操作，应当留下公开记录。
2. 影响自动化构建、发布、检查、部署的修改，应当经过 Issue 讨论和 PR 审查。
3. 维护者权限只用于维护项目，不应用于回避讨论、跳过审查或覆盖贡献者意见。
4. 遇到紧急情况可以先处置风险，但事后必须补充说明、关联 Issue，并解释为什么需要紧急处理。

## 发行版管理

1. 撤销、删除、隐藏、下架或重新发布已经公开的发行版前，必须先创建 Issue 说明原因、影响范围和计划处理方式。
2. 撤销发行版时，必须在 Issue 中通知该发行版的发布者；若发布者账号可被提及，应当直接 `@` 对方。
3. Issue 中应至少说明：
   - 被撤销的版本号或 tag；
   - 撤销原因；
   - 是否影响已下载用户；
   - 是否提供替代版本或修复版本；
   - 后续处理负责人。
4. 非安全事故、严重构建错误、法律风险或明显错误发布，不应撤销已经公开的发行版。能通过补发说明、追加修复版本、标记 pre-release 或发布新版本解决的，不应优先撤销发行版。
5. 撤销后应在原 Issue 中补充处理结果，避免只在 Releases 页面留下不可追溯的变化。

## Issue 管理

1. 非必要不关闭、不删除、不撤回、不隐藏 Issue。Issue 是项目决策和问题追踪记录的一部分。
2. 对重复、无效、无法复现或已过期的 Issue，应优先使用关闭、加标签、回复说明等方式处理。
3. 只有在以下情况才可隐藏、删除或请求平台处理 Issue：
   - 泄露隐私、密钥、账号凭据等敏感信息；
   - 包含违法、侵权、骚扰、人身攻击、垃圾广告等内容；
   - 明显滥用 Issue 系统，且普通关闭或锁定无法处理。
4. 锁定 Issue 前应尽量说明原因。因冲突、刷屏或离题讨论而锁定的，应保留原问题本身的处理结论。
5. 与维护决策有关的 Issue 不应因为结论不一致或讨论不利于某个维护者而被删除。

## 工作流与自动化修改

1. 修改 `.github/workflows/`、发布脚本、构建脚本、依赖安装流程、权限配置、自动部署配置等自动化相关代码，必须先经过 Issue 讨论。
2. 工作流修改必须通过 PR 合并，禁止直接 push 到受保护分支或默认分支。
3. PR 中应关联对应 Issue，并说明修改目的、风险范围和验证结果。
4. 涉及发布权限、仓库写权限、密钥、token、第三方服务凭据或构建产物上传的修改，应至少由一名非提交者维护者审查。
5. 紧急修复工作流时，可以先合并最小修复，但必须在 PR 或后续 Issue 中补充复盘，说明紧急原因、验证方式和是否需要追加清理。

## 直接 Push 限制

1. 禁止直接 push 修改工作流、发布流程、权限配置和安全相关代码。
2. 默认分支上的直接 push 只应用于低风险维护操作，例如修正文档错字、更新链接、同步元数据等。
3. 即使是低风险直接 push，也应在提交信息中清楚说明修改内容；有争议或影响范围不明确时应改走 PR。
4. 不得通过直接 push 绕过失败的检查、未解决的审查意见或尚未达成共识的 Issue。

## 例外与追溯

1. 若出现安全事故、恶意发布、凭据泄露、违法内容或构建产物严重损坏，维护者可以先采取必要措施降低风险。
2. 紧急处理后，应在 24 小时内补充 Issue 或 PR 记录，说明：
   - 发生了什么；
   - 为什么必须先处理；
   - 谁执行了操作；
   - 后续如何避免再次发生。
3. 例外不能成为常规流程。多次使用例外处理同类问题时，应改进工作流或权限配置。

## 执行

1. 维护者发现违反本守则的操作，应在相关 Issue、PR 或维护者讨论中指出，并要求补充记录或回滚不当修改。
2. 对项目有重大影响的维护争议，应优先通过 Issue 或 Discussion 公开讨论；不适合公开的信息可私下沟通，但应公开最终结论。
3. 本守则的修改同样应通过 Issue 讨论和 PR 合并。
